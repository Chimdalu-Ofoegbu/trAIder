# trAIder Migrations

Single Alembic migration tree managing two Postgres schemas in the `traider` database:

- **`orchestrator`** — source of truth (sessions, trades, journal entries, NAV snapshots, …)
- **`backend`** — materialized views + ephemeral tables (refreshed by triggers)

## Single-Tree Rationale (D-22 / Assumption A1)

Alembic has **no native multi-schema support** (confirmed upstream: feature gap).
The design uses one `alembic upgrade head` to manage both schemas:

- `env.py` creates both schemas (`CREATE SCHEMA IF NOT EXISTS`) before any migration runs.
- Every `op.create_table` / `op.execute` carries an explicit `schema=` argument.
- A **single `alembic_version` table** lives in the `orchestrator` schema
  (`version_table_schema="orchestrator"`) — one history table controls the entire tree.
- `include_schemas=True` in `context.configure` allows schema-qualified ops.
- `target_metadata=None` — autogenerate is **intentionally disabled**.

### Why `target_metadata=None` (no autogenerate)?

Postgres-specific objects that Alembic autogenerate **cannot represent**:
- `BRIN` indexes (only `btree` autogenerated)
- `MATERIALIZED VIEW`s
- Custom `ENUM` types
- `TRIGGER`s and trigger functions

All of these are written by hand via `op.execute()` in each migration file.
Never run `alembic revision --autogenerate` for this project.

## Developer Workflow

### Prerequisites

Docker must be running with the dev stack (`make up`).
The `traider` database is created automatically by the Postgres container.

### Running Migrations

```bash
# Full upgrade to latest schema (idempotent — safe to run repeatedly)
make db-migrate

# Or directly:
export DATABASE_URL=postgresql+psycopg://migrator_user:migrator_pass@localhost:5432/traider
uv run alembic -c migrations/alembic.ini upgrade head

# Generate SQL DDL without a live DB (offline audit):
uv run alembic -c migrations/alembic.ini upgrade head --sql

# Downgrade to base (wipes all trAIder schemas — dev reset only):
uv run alembic -c migrations/alembic.ini downgrade base
```

### Clean Rebuild (`make db-reset`)

```bash
make reset
# Equivalent to:
#   make down        → stop + remove containers + volumes
#   make up          → start postgres + redis + anvil + pgadmin (with healthchecks)
#   make seed        → run alembic upgrade head + seed USDC/ETH balances
#   make verify-stack → assert schemas/tables/ENUM/roles/balances correct
```

### Adding a New Migration

```bash
# Create a new migration file (always hand-written, never --autogenerate):
uv run alembic -c migrations/alembic.ini revision -m "description_of_change"
# Edit the generated migrations/versions/{rev}_description_of_change.py
# Follow the rules in script.py.mako header comments.
```

**Migration naming convention:** `{NNNN}_{description}.py` (zero-padded 4-digit prefix).

## Database URL

The migration runner reads `DATABASE_URL` from the environment. **Never commit credentials.**

```bash
# Local dev (matches docker-compose.yml):
export DATABASE_URL=postgresql+psycopg://migrator_user:migrator_pass@localhost:5432/traider

# CI (set as a GitHub Actions secret):
# DATABASE_URL=postgresql+psycopg://migrator_user:${MIGRATOR_PASS}@localhost:5432/traider
```

The URL **must use a synchronous driver** (`psycopg` or `psycopg2`) for Alembic.
Runtime services (`orchestrator`, `backend`) use `asyncpg` for their connection pools.

## Schema Design

### `orchestrator` schema — source of truth

| Table | Purpose |
|-------|---------|
| `sessions` | 72-hour trading session metadata |
| `vaults` | Per-model ERC-4626 vault addresses |
| `positions` | Open perpetual positions per vault |
| `trades` | Executed trades (immutable, append-only) |
| `journal_entries` | Three-phase commit state machine (D-21) |
| `model_decisions` | Raw LLM decision payloads per cycle |
| `nav_snapshots` | NAV time-series (append-only, BRIN indexed) |
| `journal_state_log` | Append-only transition history (debug) |
| `model_status_log` | Paused/malformed model states (ORCH-06) |
| `event_log` | Structured operational logs (D-69) |

### `backend` schema — read-optimized layer

| Table / View | Type | Purpose |
|--------------|------|---------|
| `dashboard_model_state` | MATERIALIZED VIEW | Latest NAV + position per vault, refreshed by trigger |
| `dashboard_session_state` | MATERIALIZED VIEW | Session-level aggregates |
| `websocket_sessions` | Table | Active WebSocket connections (ephemeral) |
| `verifier_replay_log` | Table | Verifier CLI replay results |

### Journal State Machine (D-21)

```
pending_pin → pinned_primary → pinned_backup → signed → submitted → recorded
                                                                  ↘ failed
```

**Critical:** `submitted` persists `onchain_tx` hash **before** broadcast.
Recovery: query chain status before resubmit. Never blindly resubmit a `submitted` entry.

## Role Privilege Matrix (D-19)

| Role | orchestrator.* | backend.* | Notes |
|------|---------------|-----------|-------|
| `orchestrator_user` | SELECT, INSERT, UPDATE, DELETE | **NO ACCESS** | Orchestrator service runtime |
| `backend_user` | SELECT (read-only) | SELECT, INSERT, UPDATE, DELETE | Backend service runtime |
| `migrator_user` | DDL (via alembic) | DDL (via alembic) | Alembic only; never used by running services |

A compromised `backend_user` cannot tamper with source-of-truth `orchestrator.*` tables.

## Indexes

- **BRIN** on `(vault_address, block_timestamp)` for `orchestrator.nav_snapshots` and `orchestrator.trades`
  — cheap range scans over append-only time-series data (D-20).
- **B-tree** on `orchestrator.trades(trade_hash)` — fast journal lookup by trade hash.

## Testing

```bash
# Integration tests (require live Postgres via TEST_DATABASE_URL):
export TEST_DATABASE_URL=postgresql+psycopg://migrator_user:pass@localhost:5432/traider_test
uv run pytest migrations/tests/test_migrations.py -v

# Without a DB, tests skip gracefully (pytest.skip):
uv run pytest migrations/tests/test_migrations.py -v
# → SKIPPED: No Postgres reachable (TEST_DATABASE_URL not set or DB unavailable)
```

Live upgrade/downgrade integration tests are authoritative validation.
In Docker-less environments, use `alembic upgrade head --sql` to audit the DDL offline.
