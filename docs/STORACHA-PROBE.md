## Decision

**DECISION: filebase**

**Rationale:**

1. The Storacha w3up endpoint (`https://up.web3.storage`) is reachable and running v3.0.0,
   but requires UCAN capability delegation — a non-trivial auth flow that requires
   creating a Space, delegating capabilities, and POSTing CAR files. No simple
   HTTP Bearer token path exists for Python without the full w3up-client library
   (JavaScript-only as of 2026-06-06; Python would need direct UCANTO CBOR encoding).

2. The legacy `api.web3.storage` endpoint is in maintenance and deprecated.

3. **Filebase** (S3-compatible API → IPFS) is the recommended fallback per 03-RESEARCH.md
   Pattern 8 Option B. It exposes a standard S3-compatible PUT endpoint that accepts
   a Bearer API key with no custom auth flow — much simpler to call from Python via httpx.
   No API key is configured yet, but the integration path is straightforward.

**Consequence for Wave 2 (03-06):**

- `pin_to_storacha_backup()` will use the **Filebase S3-compatible API**
  (endpoint: `https://s3.filebase.com`, PUT `/{bucket}/{key}`, Bearer auth).
- The operator must provision a Filebase account and set `FILEBASE_API_KEY`
  and `FILEBASE_BUCKET` in `.env` before Wave 2 (03-06) lands.
- If Filebase is also unavailable at Wave 2 probe time, the fallback is:
  skip the backup pin, log CRITICAL alert, continue with Pinata-only.
  Dual-pin parity check (JOURNAL-02) is non-fatal if backup is unavailable
  (single-pin degraded mode, documented in JOURNAL-02 acceptance criteria).
